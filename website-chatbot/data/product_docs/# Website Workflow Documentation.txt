# Website Workflow Documentation

## Overview
[cite_start]This document outlines the complete workflow of the **AI-Insight-Pro** platform[cite: 227]. [cite_start]The system is designed to help organizations identify, classify, and protect sensitive data elements (SDEs) across their infrastructure through comprehensive data source scanning and risk analysis[cite: 228].

## Frontend Workflow

### 1. Login
#### User Authentication & Onboarding
- **Entry Point**: User visits the application homepage
- **Registration**: New users must provide:
  - Full Name (for identification)
  - Username (unique identifier)
  - Company Name (for industry-specific analysis)
  - Email Address (for communication)
  - Password (secure access)
#### User Session Management
- **Unique Key Generation**: Upon successful login, a unique session key is created
- **Session Tracking**: User activities are tracked with their unique identifier
- **Multi-session Support**: Users can access from multiple devices with same credentials
#### Post-Login Navigation
- **Dashboard**: Welcome screen with user-specific information
- [cite_start]**Navbar Menu**: Persistent navigation with multiple options [cite: 230]
- **User Profile**: Access to account settings and preferences

### 2. SDE Selection - Core Feature
#### What are SDEs (Sensitive Data Elements)?
[cite_start]**SDEs** are specific types of data that require special protection due to their sensitive nature[cite: 231]. They include:
- **Personal Identifiable Information (PII)**:
  - Social Security Numbers
  - Credit Card Numbers
  - Driver's License Numbers
  - Passport Numbers
  - Email Addresses
  - Phone Numbers
- **Financial Information**:
  - Bank Account Numbers
  - Credit Card Details
  - Tax Identification Numbers
  - Salary Information
- **Healthcare Data**:
  - Medical Record Numbers
  - Health Insurance Numbers
  - Patient Identifiers
  - Diagnosis Codes
- **Business Sensitive Data**:
  - Trade Secrets
  - Intellectual Property
  - Customer Lists
  - Financial Reports
#### Why SDEs are Critical
1. [cite_start]**Compliance Requirements**: GDPR, CCPA, HIPAA, SOX regulations [cite: 233]
2. **Data Breach Prevention**: Identifying sensitive data helps prevent leaks
3. **Risk Assessment**: Understanding data sensitivity enables better risk management
4. **Privacy Protection**: Ensures customer and employee privacy
5. **Legal Obligations**: Many industries have mandatory data protection requirements
#### SDE Catalogue Features
- **Industry-Specific SDEs**:
  - **Healthcare**: HIPAA-compliant data elements
  - **Finance**: PCI-DSS and SOX requirements
  - **Retail**: Customer PII and payment data
  - **Technology**: Intellectual property and trade secrets
  - **Government**: Classified and sensitive information
- **SDE Management**:
  - **Browse Existing SDEs**: View industry-specific sensitive data elements
  - **Add New SDEs**: Create custom SDEs with regex patterns
  - **Edit SDEs**: Modify existing patterns and descriptions
  - [cite_start]**Delete SDEs**: Remove obsolete or incorrect entries [cite: 234]
  - **Search & Filter**: Find specific SDEs by category or name
- **Regex Pattern Management**:
  - **Pattern Creation**: Define regex patterns for new SDEs
  - **Pattern Testing**: Validate regex against sample data
  - **Pattern Documentation**: Add descriptions and usage notes
  - **Version Control**: Track changes to SDE patterns
#### SDE Workflow Process
1. **Discovery**: System scans data sources for potential SDEs
2. **Classification**: Identified data is classified by type
3. **Validation**: Human review of automated classifications
4. **Documentation**: SDEs are documented with patterns and descriptions
5. **Monitoring**: Continuous scanning for new SDE instances
6. **Reporting**: Generate reports on SDE distribution and risk

### 3. Model Registry
- **Overview**: This section allows users to manage and register their custom AI models.
- **Model Upload**: Users can upload their own model weights and configurations.
- **Model Selection**: Choose which models to use for SDE detection and classification.

### 4. Data Connection - Connect Navbar Option
#### Data Source Types Supported
- [cite_start]**Cloud Databases**: BigQuery, PostgreSQL, MySQL [cite: 235]
- **Cloud Storage**: GCP Buckets, AWS S3
- **On-Premise Databases**: Local PostgreSQL, MySQL servers
- **File Systems**: Local and network file storage
#### Connection Process
1. **Source Selection**: Choose data source type from dropdown
2. **Credential Entry**: Provide connection credentials
   - **For BigQuery**: Project ID, Dataset ID, Service Account JSON
   - **For PostgreSQL/MySQL**: Host, Port, Database, Username, Password
   - **For GCP Buckets**: Project ID, Bucket Name, Service Account JSON
3. **Connection Testing**: Validate credentials before proceeding
4. **Registration**: Successfully connected sources are registered in the system
#### Credential Security
- [cite_start]**Encrypted Storage**: All credentials are encrypted at rest [cite: 236]
- **Secure Transmission**: HTTPS encryption for all data transfers
- **Access Control**: Role-based access to connection details
- **Audit Logging**: Track all connection attempts and usage
